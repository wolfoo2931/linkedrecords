## Overview

Most applications should use the [Browser SDK](/getting-started) or the
[React hooks](/react), which handle authentication, real-time subscriptions,
and CRDT/OT change encoding for you. This reference documents the underlying
HTTP endpoints for building custom clients or debugging.

<Callout type="warn">
Record **updates** (`PATCH`) do not accept plain values - they expect
CRDT/OT change payloads produced by the SDK's data type implementations.
Reading records and managing facts works well over plain HTTP; for updating
record values, use the SDK.
</Callout>

## Authentication

All API requests require authentication via:

- **Cookie-based auth** (confidential client mode)
- **Bearer token** (public client mode, requires `ALLOW_HTTP_AUTHENTICATION_HEADER=true`)

```bash
# Bearer token
curl -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  https://your-linkedrecords-server.com/facts
```

## Base URL

All endpoints are relative to your LinkedRecords server URL:

```
https://your-linkedrecords-server.com
```

<Callout type="info">
The `/records` and `/record-compositions` routes are also mounted under their
legacy names `/attributes` and `/attribute-compositions`. Both forms hit the
same handlers; new integrations should use the `/records` form.
</Callout>

## Records

### Get Record

Retrieve a single record by ID.

```
GET /records/:id?clientId=<clientId>
```

**Parameters:**
- `id` - Record ID. The prefix encodes the type: `kv-` (key-value), `l-` (long text), `bl-` (blob)
- `clientId` - A client-generated identifier for this connection (query parameter, required)
- `valueOnly=true` - Optional; respond with the raw value only instead of the serialized record state

**Response** contains the serialized record state, including the current `value`.
Blob values are returned as `data:<mimetype>;base64,...` URIs.

### Query Records

Find records matching a query. Each key in the query object names a result
set; the values are arrays of fact criteria (see the
[query language](/query-language)).

```
GET /records?query=<url-encoded JSON>
```

**Query format (before URL encoding):**
```json
{
  "docs": [["$it", "isA", "Document"]],
  "tasks": [["$it", "isA", "Task"]]
}
```

Data type filters use the legacy type names as strings:
`["$it", "$hasDataType", "KeyValueAttribute"]` (also `LongTextAttribute`,
`BlobAttribute`).

**Response:** an object with the same keys, each holding an array of
serialized records:

```json
{
  "docs": [
    { "id": "kv-abc123", "value": { "title": "..." }, "readToken": "..." }
  ],
  "tasks": []
}
```

### Create Record

Create a new record. The record type is selected with the `dtp` (data type
prefix) query parameter.

```
POST /records?dtp=<kv|l|bl>&clientId=<clientId>
```

**Parameters:**
- `dtp` - Data type prefix: `kv` (key-value), `l` (long text), `bl` (blob)
- `clientId` - Client identifier (query parameter, required)

**Body (key-value and long text):**
```json
{
  "clientId": "<clientId>",
  "actorId": "us-...",
  "value": { "title": "New Document" },
  "facts": [
    ["$it", "isA", "Document"]
  ]
}
```

For **blob** records, send `multipart/form-data` with the binary content in a
field named `change`.

Facts must have `$it` as subject or object (two-element facts
`["isA", "Document"]` are shorthand for `["$it", "isA", "Document"]`); other
facts are ignored here - create them via `POST /facts` instead.

**Response:** the serialized record state including its `id`.

**Error:** if any submitted fact is not authorized, the record is not created
and the server responds `403` with:
```json
{ "unauthorizedFacts": [ ... ] }
```

### Update Record

Apply a change to a record.

```
PATCH /records/:id?clientId=<clientId>
```

The body is a `SerializedChangeWithMetadata` - the CRDT (key-value) or
Operational Transform (long text) change format produced by the SDK. It is
not a plain `{ "value": ... }` replacement. Use the SDK for updates.

**Response:** `200` with an empty body. The committed change is broadcast to
all subscribed clients.

### Create Multiple Records (Blueprint)

Create multiple related records atomically. `{{name}}` placeholders in facts
refer to other records in the same request.

```
POST /record-compositions?clientId=<clientId>
```

**Body:**
```json
{
  "parent": {
    "type": "KeyValueAttribute",
    "value": { "name": "Parent" },
    "facts": [["$it", "isA", "Parent"]]
  },
  "child": {
    "type": "KeyValueAttribute",
    "value": { "name": "Child" },
    "facts": [
      ["$it", "isA", "Child"],
      ["{{parent}}", "$isAccountableFor", "$it"]
    ]
  }
}
```

The `type` field uses the legacy type names `KeyValueAttribute`,
`LongTextAttribute`, or `BlobAttribute`; it defaults to `KeyValueAttribute`
when omitted. Every entry must include a `value`.

**Response:**
```json
{
  "parent": { "id": "kv-parent123", "value": { "name": "Parent" } },
  "child": { "id": "kv-child456", "value": { "name": "Child" } }
}
```

**Error:** if any fact is not authorized, nothing is created and the server
responds `403` with `{ "unauthorizedFacts": [ ... ] }`.

## Facts

### Query Facts

Find facts matching criteria. Each of `subject`, `predicate`, and `object` is
an optional JSON-encoded array of criteria.

```
GET /facts?subject=<JSON>&predicate=<JSON>&object=<JSON>
```

**Example:**
```bash
# Find all term definitions
curl "https://your-server.com/facts?predicate=%5B%22%24isATermFor%22%5D"
```

**Response:**
```json
[
  {
    "subject": "Document",
    "predicate": "$isATermFor",
    "object": "A file or record"
  }
]
```

### Create Facts

Create new facts. The body is a **plain JSON array** of `[subject, predicate,
object]` triples (not wrapped in an object).

```
POST /facts
```

**Body:**
```json
[
  ["Document", "$isATermFor", "A file or record"],
  ["us-user123", "$isMemberOf", "kv-team456"]
]
```

**Response:** `200` with the array of created facts.

<Callout type="warn">
Authorization is all-or-nothing per request: if **any** fact in the batch is
not authorized, **no** facts are saved and the server responds `403` with an
empty object. The SDK's `Fact.createAll()` resolves to an empty array in that
case - check the return value.
</Callout>

### Delete Facts

Remove existing facts. Note this is a `POST` to `/facts/delete`, not an HTTP
`DELETE`. The body is a plain JSON array of triples.

```
POST /facts/delete
```

**Body:**
```json
[
  ["us-user123", "$isMemberOf", "kv-team456"]
]
```

**Response:** `200` with `{ "status": "ok" }`, or `403` if any fact in the
batch is not authorized (nothing is deleted in that case).

## User Information

### Get User Info

Get the current user's information.

```
GET /userinfo
```

**Response:**
```json
{
  "userId": "us-abc123",
  "userPicture": "https://...",
  "userEmail": "user@example.com"
}
```

### Look Up User by Email

Find a user's ID by their email address.

```
GET /userinfo?email=user@example.com
```

**Response:** `{ "id": "us-abc123" }`, or `{}` when no user with that email
is known.

## Quota

### Get Quota

Get storage quota for a user or organization. Only the accountable user (or
members of the accountee) can read it.

```
GET /quota/:nodeId
```

**Parameters:**
- `nodeId` - User ID or organization record ID

**Response** (sizes in bytes; `{}` if you are not authorized to read it):
```json
{
  "nodeId": "us-abc123",
  "userIsAccountable": true,
  "isUpgraded": false,
  "totalStorageAvailable": 524288000,
  "usedStorage": 1048576,
  "remainingStorageAvailable": 523239424,
  "accounteeInformation": null
}
```

## Members

### Get Team Members

Get the users that are members, hosts, or accountable for a record. Only the
record's creator or a host can read this.

```
GET /records/:id/members?clientId=<clientId>
```

**Response:**
```json
[
  { "id": "us-123", "username": "alice@example.com" },
  { "id": "us-456", "username": "bob@example.com" }
]
```

**Response when not authorized:**
```json
{ "notVisibleToUser": true }
```

## OIDC Discovery

### Get OIDC Configuration

Get the OIDC provider configuration the server is using, so public clients
can configure their login flow. This endpoint does not require
authentication.

```
GET /oidc/discovery
```

**Response:**
```json
{
  "issuer": "https://auth.example.com/",
  "authorization_endpoint": "https://auth.example.com/authorize",
  "token_endpoint": "https://auth.example.com/oauth/token",
  "userinfo_endpoint": "https://auth.example.com/userinfo"
}
```

## Real-Time Updates

Real-time updates are delivered over Socket.IO. The SDK manages these
connections automatically (per-record change channels and query
subscriptions); the wire protocol is internal and may change between
versions. If you build a custom client, use the SDK source
(`lib/client-server-bus/`) as the reference.

---

## Error Responses

| Status | Meaning |
|--------|---------|
| `401` | Not authenticated or session expired. |
| `403` | Unauthorized facts in the request (response contains `{ "unauthorizedFacts": [...] }` for record creation, `{}` for fact endpoints), or not enough storage quota (`Not enough storage space available`). |
| `404` | Record not found, not accessible, or unknown record ID prefix. |
| `500` | Unexpected server-side error. |

---

## Rate Limiting

The server limits each IP to 1000 requests per second
(see `src/server/routes.ts`). Deployments behind proxies or gateways may
impose additional limits.

---

## CORS

For cross-origin requests, ensure your origin is included in the `CORS_ORIGIN`
environment variable configuration on the server:

```bash
CORS_ORIGIN='["https://your-app.com"]'
```
